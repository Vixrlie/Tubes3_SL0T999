\section{Analisis Pemecahan Masalah}

Permasalahan pada spesifikasi dipecahkan menjadi rancangan solusi sesuai implementasi pada codebase ini. Cakupan analisis berupa alur pemindaian, pemetaan algoritma pencocokan, dan visualisasi hasil.

\subsection{Formulasi Masalah dan Strategi Umum}

Permasalahan utama pada proyek ini bukan hanya mencari kata kunci dari \texttt{keywords.txt}, tetapi juga menjaga agar sistem tetap relevan ketika pola ditulis dengan variasi yang sengaja menyamarkan bentuk aslinya. Pada praktiknya, halaman web juga bersifat dinamis karena DOM dapat berubah setelah halaman dimuat, sehingga proses deteksi harus mampu melakukan pemindaian ulang tanpa mengganggu pengalaman pengguna.

Jika masalah tersebut langsung diselesaikan dengan satu engine exact matching, hasilnya akan cepat tetapi rapuh terhadap manipulasi karakter. Sebaliknya, jika seluruh token selalu diproses dengan fuzzy matching, sistem menjadi lebih mahal dari sisi waktu eksekusi dan lebih berisiko menghasilkan false positive. Karena itu, arsitektur yang dipilih bersifat berlapis. Lapisan pertama memprioritaskan exact matching yang deterministik, lapisan kedua menangani pola struktural melalui RegEx, dan lapisan ketiga dipakai untuk fuzzy matching sebagai fallback terkontrol.

Di sisi antarmuka, hasil deteksi tidak cukup hanya dihitung. Sistem juga perlu menunjuk elemen yang terdeteksi secara presisi, menampilkan alasan deteksi lewat tooltip, serta menyajikan statistik yang konsisten pada popup. Kebutuhan ini membuat pemecahan masalah tidak berhenti pada algoritma string matching, tetapi berlanjut ke desain alur data antar \textit{content script}, \textit{popup}, dan \textit{background service worker}.

\subsection{Pemetaan Masalah ke Pipeline Implementasi}

Pada implementasi saat ini, alur utama dimulai saat \textit{content script} membangun \texttt{ScanRequest} dan \texttt{ContentPipelineState}. Tahap ini mengumpulkan target teks dari DOM melalui \texttt{TreeWalker}, lalu mengumpulkan target gambar melalui \texttt{querySelectorAll('img')}. Elemen yang tidak relevan, tidak terlihat, atau termasuk UI internal extension disaring agar hasil deteksi tidak bias.

Setelah konteks siap, modul \texttt{runDetectionEnginesDetailed()} mengeksekusi engine secara berurutan. Urutan ini penting karena hasil dari exact matching dipakai untuk menekan ruang pencarian fuzzy matching. Secara konseptual, pipeline ini dapat dipandang sebagai bentuk optimasi heuristik. Sistem mencari jawaban paling pasti terlebih dahulu, baru membuka toleransi pada tahap berikutnya.

\subsection{Pemetaan ke Elemen Algoritma KMP dan Boyer-Moore}

KMP pada proyek ini dipetakan ke dua komponen kunci, yaitu pembangunan \textit{failure function} dan proses \textit{scan} linear pada teks target. Tujuan dari pemetaan ini adalah menghindari pemeriksaan ulang karakter yang sudah pasti tidak perlu dibandingkan lagi. Saat mismatch terjadi, indeks pola bergerak mengikuti nilai \textit{failure function}, sedangkan indeks teks tetap maju. Pola tersebut sesuai dengan teori KMP yang dibahas pada Bab 2.

Berikut adalah cuplikan inti implementasi KMP yang digunakan:

\begin{lstlisting}[language={}, breaklines=true, breakatwhitespace=false, columns=fullflexible, caption={Cuplikan inti KMP pada file src/algorithms/kmp.ts}]
while (textIndex < text.length) {
  comparisons += 1;
  if (text[textIndex] === pattern[patternIndex]) {
    textIndex += 1;
    patternIndex += 1;
    if (patternIndex === pattern.length) {
      matches.push(textIndex - patternIndex);
      patternIndex = failure[patternIndex - 1];
    }
    continue;
  }

  if (patternIndex !== 0) {
    patternIndex = failure[patternIndex - 1];
    continue;
  }

  textIndex += 1;
}
\end{lstlisting}

Boyer-Moore dipetakan ke \textit{last occurrence table} dan \textit{good suffix table}. Perbandingan dimulai dari sisi kanan pola, kemudian besar pergeseran ditentukan dari maksimum dua heuristik tersebut. Dengan strategi ini, sistem dapat melompati alignment yang jelas tidak mungkin cocok.

\begin{lstlisting}[language={}, breaklines=true, breakatwhitespace=false, columns=fullflexible, caption={Cuplikan penentuan pergeseran Boyer-Moore pada file src/algorithms/boyerMoore.ts}]
const lastIndex = lastOccurrence.get(text[shift + index]);
const badCharShift = index - (lastIndex ?? -1);
const goodSuffixShift = goodSuffix[index + 1];
shift += Math.max(badCharShift, goodSuffixShift);
\end{lstlisting}

Secara analitis, dua cuplikan di atas memperlihatkan bahwa proses pemetaan dari teori ke kode tidak berhenti pada nama algoritma, tetapi benar-benar menerapkan mekanisme inti yang diwajibkan, termasuk \textit{failure function}, \textit{last occurrence}, dan \textit{shifting process}.

\subsection{RegEx dan Fuzzy sebagai Lapisan Adaptif}

Setelah exact matching dijalankan, sistem menambahkan lapisan RegEx untuk menangkap bentuk token umum yang mengikuti pola kata diikuti 2 atau 3 digit. Pola yang dipakai pada implementasi adalah \verb|[\p{L}][\p{L}\p{M}\p{N}'._-]*\d{2,3}|. Secara praktis, pola ini memungkinkan sistem tetap sensitif terhadap variasi penulisan tanpa bergantung pada daftar keyword statis saja.

Fuzzy matching kemudian dipakai sebagai lapisan adaptif untuk karakter visual-similar. Implementasi weighted Levenshtein memberi bobot substitusi rendah untuk pasangan seperti \texttt{o-0}, \texttt{i-1}, dan \texttt{a-4}. Selain itu, skor dinormalisasi dengan panjang token agar threshold konsisten lintas kata pendek maupun panjang.

\begin{lstlisting}[language={}, breaklines=true, breakatwhitespace=false, columns=fullflexible, caption={Cuplikan normalisasi skor fuzzy pada file src/algorithms/weightedLevenshtein.ts}]
const distance = weightedLevenshtein(tokenLower, normalizedKeyword);
const score = distance / Math.max(tokenLower.length, normalizedKeyword.length);
if (score <= DEFAULT_THRESHOLD) {
  // kandidat match fuzzy
}
\end{lstlisting}

Pendekatan ini menjaga kompromi antara recall dan precision. Sistem tetap cukup toleran terhadap manipulasi ringan, tetapi tidak langsung menandai semua token yang hanya mirip sebagian kecil.

\subsection{Arsitektur Fungsional Extension}

\subsection{Ilustrasi Kasus dan Pembacaan Hasil}

Sebagai ilustrasi, misalkan sebuah halaman memuat teks ``H0KI88 lagi promo'' dan ``MAXWIN234'', serta satu gambar yang setelah OCR terbaca ``GAC0R99''. Pada kasus ini, token \texttt{MAXWIN234} akan cepat tertangkap oleh RegEx karena mengikuti pola struktural kata+angka. Token \texttt{H0KI88} berpotensi tidak lolos exact matching, tetapi masih dapat ditandai oleh fuzzy matching karena jarak edit berbobotnya berada di bawah threshold. Untuk teks hasil OCR, sistem menjalankan logika deteksi yang sama setelah proses ekstraksi teks gambar selesai.

Ketika match ditemukan, setiap entri hasil membawa \texttt{targetIndex}. Informasi ini membuat sistem dapat menunjuk elemen sumber dengan presisi saat menerapkan highlight dan tooltip. Dengan demikian, pengguna tidak hanya melihat total temuan, tetapi juga tahu elemen mana yang terdeteksi, algoritma apa yang memicu deteksi, berapa jumlah kemunculannya, dan berapa waktu eksekusinya.

\subsection{Analisis Kesesuaian dengan Spesifikasi}

Jika dibandingkan dengan spesifikasi wajib, implementasi saat ini sudah memenuhi kebutuhan inti secara konsisten. Exact matching dari \texttt{keywords.txt} dijalankan oleh KMP dan Boyer-Moore from scratch, lalu diperkuat dengan Aho-Corasick dan Rabin-Karp pada jalur bonus. RegEx menangani pola kata+angka, sedangkan weighted Levenshtein menangani variasi visual-similar. Dari sisi antarmuka, elemen terdeteksi dapat ditandai, tooltip ditampilkan saat hover, dan statistik real-time tersaji di popup. Fitur blur serta OCR juga telah diintegrasikan ke pipeline yang sama sehingga tidak menjadi modul terpisah yang inkonsisten.

Secara keseluruhan, proses pemecahan masalah pada proyek ini menunjukkan bahwa keputusan desain algoritmik, pengolahan DOM, dan arsitektur extension saling mendukung. Hasil akhirnya bukan sekadar demonstrasi algoritma, tetapi sistem deteksi yang operasional pada konteks halaman web nyata.
